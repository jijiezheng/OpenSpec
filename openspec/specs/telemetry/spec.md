# 遥测规范

## 目的

本规范定义了 OpenSpec 如何收集匿名使用遥测数据以帮助改进工具。它管辖 `src/telemetry/` 模块，该模块处理 PostHog 集成、隐私保护事件设计、用户选择退出机制以及首次运行通知显示。本规范确保遥测数据最小化、透明化，并尊重用户隐私。

## 需求

### 需求：命令执行追踪
系统应在任何 CLI 命令执行时向 PostHog 发送 `command_executed` 事件，仅包含命令名称和 OpenSpec 版本作为属性。

#### 场景：标准命令执行
- **当** 用户运行任何 openspec 命令
- **那么** 系统发送带有 `command` 和 `version` 属性的 `command_executed` 事件

#### 场景：子命令执行
- **当** 用户运行嵌套命令如 `openspec change apply`
- **那么** 系统发送带有完整命令路径的 `command_executed` 事件（例如 `change:apply`）

### 需求：隐私保护事件设计
系统不得在遥测事件中包含命令参数、文件路径、项目名称、规范内容、错误消息或 IP 地址。

#### 场景：带参数的命令
- **当** 用户运行 `openspec init my-project --force`
- **那么** 遥测事件仅包含 `command: "init"` 和 `version: "<version>"`，不包含参数

#### 场景：IP 地址排除
- **当** 系统发送遥测事件时
- **那么** 事件明确设置 `$ip: null` 以防止 IP 追踪

### 需求：环境变量选择退出
当设置 `OPENSPEC_TELEMETRY=0` 或 `DO_NOT_TRACK=1` 环境变量时，系统应禁用遥测。

#### 场景：OPENSPEC_TELEMETRY 选择退出
- **当** 环境中设置了 `OPENSPEC_TELEMETRY=0`
- **那么** 系统不发送任何遥测事件

#### 场景：DO_NOT_TRACK 选择退出
- **当** 环境中设置了 `DO_NOT_TRACK=1`
- **那么** 系统不发送任何遥测事件

#### 场景：环境变量优先
- **当** 用户之前使用过 CLI（配置存在）
- **并且** 用户设置了 `OPENSPEC_TELEMETRY=0`
- **那么** 无论配置状态如何，遥测都被禁用

### 需求：CI 环境自动禁用
当检测到 `CI=true` 环境变量时，系统应自动禁用遥测。

#### 场景：CI 环境检测
- **当** 环境中设置了 `CI=true`
- **那么** 系统不发送任何遥测事件

#### 场景：CI 与显式启用
- **当** 设置了 `CI=true`
- **并且** 显式设置了 `OPENSPEC_TELEMETRY=1`
- **那么** 遥测保持禁用状态（CI 优先以保护隐私）

### 需求：首次运行遥测通知
系统应在首次命令执行时显示一行遥测披露通知，在发送任何遥测数据之前。

#### 场景：首次命令执行
- **当** 用户运行第一个 openspec 命令
- **并且** 遥测已启用
- **那么** 系统显示："注意：OpenSpec 收集匿名使用统计。退出选项：OPENSPEC_TELEMETRY=0"

#### 场景：后续命令执行
- **当** 用户已看过通知（配置中 `noticeSeen: true`）
- **那么** 系统不显示通知

#### 场景：通知在遥测之前
- **当** 显示首次运行通知时
- **那么** 通知在任何遥测事件发送之前出现

### 需求：匿名用户识别
系统应在首次遥测发送时生成随机 UUID 作为匿名标识符，存储在全局配置中。

#### 场景：首次遥测事件
- **当** 发送第一个遥测事件
- **并且** 配置中不存在 `anonymousId`
- **那么** 系统生成随机 UUID v4 并将其存储在配置中

#### 场景：持久身份
- **当** 用户跨会话运行多个命令时
- **那么** 所有事件使用相同的 `anonymousId`

#### 场景：延迟生成与选择退出
- **当** 用户在任何命令运行之前选择退出
- **那么** 永远不会生成或存储 `anonymousId`

### 需求：即时事件发送
系统应使用 `flushAt: 1` 和 `flushInterval: 0` 配置立即发送遥测事件，不进行批处理。

#### 场景：事件传输时机
- **当** 命令执行时
- **那么** 遥测事件立即发送，不排队等待批量传输

### 需求：优雅关闭
系统应在 CLI 退出前调用 `posthog.shutdown()` 以确保待处理事件被刷新。

#### 场景：正常退出
- **当** 命令成功完成时
- **那么** 系统在退出前等待 `shutdown()`

#### 场景：错误退出
- **当** 命令因错误失败时
- **那么** 系统仍在退出前等待 `shutdown()`

### 需求：静默失败处理
系统应静默忽略遥测失败，不影响 CLI 功能。

#### 场景：网络故障
- **当** 遥测请求因网络错误失败时
- **那么** CLI 命令正常完成，不显示错误消息

#### 场景：PostHog 服务中断
- **当** PostHog 服务不可用时
- **那么** CLI 命令正常完成，不显示错误消息

#### 场景：关闭失败
- **当** `shutdown()` 失败或超时时
- **那么** CLI 正常退出，不显示错误消息
