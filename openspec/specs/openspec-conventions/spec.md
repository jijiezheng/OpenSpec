# OpenSpec 约定规范

## 目的

OpenSpec 约定应定义系统能力如何被文档化、变更如何被提案和跟踪、以及规格如何随时间演变。这个元规范作为 OpenSpec 自身约定的真实来源。

## 需求

### 需求：规格和变更的结构化约定

OpenSpec 约定应强制使用结构化的规格格式，包含清晰的 requirements 和 scenarios 部分，以便工具可以一致地解析。

#### 场景：遵循结构化规格格式

- **当** 编写或更新 OpenSpec 规格时
- **那么** 作者应使用 `### Requirement: ...` 后跟至少一个 `#### Scenario: ...` 部分

### 需求：行为优先的规格边界
OpenSpec 规格应捕获可验证的行为契约，避免内部实现细节。

#### 场景：编写行为需求

- **当** 在 `spec.md` 中记录能力时
- **那么** 需求专注于外部可观察行为、接口、错误处理和约束
- **并且** 场景保持可测试或明确可验证

#### 场景：避免实现泄露

- **当** 细节涉及具体的库选择、类/函数结构或执行机制时
- **那么** 这些细节应记录在 `design.md` 或 `tasks.md` 中，而不是行为需求中

### 需求：渐进式严谨性
OpenSpec 约定应默认保持规格轻量，仅在风险或协调复杂性需要时才增加严谨性。

#### 场景：常规变更规格

- **当** 变更是本地的且低风险时
- **那么** 作者使用简洁的、行为优先的需求，最小化仪式

#### 场景：高风险或跨边界变更规格

- **当** 变更是跨团队、跨仓库、API 契约破坏、迁移繁重或安全/隐私敏感时
- **那么** 作者按比例增加细节和明确验证期望

### 需求：项目结构
OpenSpec 项目应维护规格和变更的一致目录结构。

#### 场景：初始化项目结构

- **当** OpenSpec 项目被初始化时
- **那么** 它应具有以下结构：
```
openspec/
├── project.md              # 项目特定上下文
├── AGENTS.md               # AI 助手指令
├── specs/                  # 当前已部署的能力
│   └── [capability]/       # 单个、聚焦的能力
│       ├── spec.md         # WHAT 和 WHY
│       └── design.md       # HOW（可选，用于已建立的模式）
└── changes/                # 提议的变更
    ├── [change-name]/      # 描述性变更标识符
    │   ├── proposal.md     # Why、what 和影响
    │   ├── tasks.md        # 实现检查清单
    │   ├── design.md       # 技术决策（可选）
    │   └── specs/          # 完整的未来状态
    │       └── [capability]/
    │           └── spec.md # 干净的 markdown（无 diff 语法）
    └── archive/            # 已完成的变更
        └── YYYY-MM-DD-[name]/
```

### 需求：行为规格的结构化格式

行为规格应使用一致部分标题和关键字的结构化格式，以确保视觉一致性和可解析性。

#### 场景：编写需求部分

- **当** 在行为规格中记录需求时
- **那么** 使用格式 `### Requirement: [Name]` 的三级标题
- **并且** 立即跟随描述核心行为的 SHALL 语句
- **并且** 保持需求名称描述性且少于 50 个字符

#### 场景：记录场景

- **当** 记录特定行为或用例时
- **那么** 使用格式 `#### Scenario: [Description]` 的四级标题
- **并且** 使用带粗体关键字的项目符号表示步骤：
  - **GIVEN** 用于初始状态（可选）
  - **WHEN** 用于条件或触发器
  - **THEN** 用于预期结果
  - **AND** 用于额外结果或条件

#### 场景：添加实现细节

- **当** 步骤需要额外细节时
- **那么** 在主步骤下使用子项目符号
- **并且** 保持一致的缩进
  - 子项目符号提供示例或细节
  - 保持子项目符号简洁

### 需求：基于标题的需求标识

需求标题应作为当前规格和提议变更之间程序化匹配的唯标识符。

#### 场景：程序化匹配需求

- **当** 处理增量变更时
- **那么** 使用 `### Requirement: [Name]` 标题作为唯标识符
- **并且** 使用规范化标题进行匹配：`normalize(header) = trim(header)`
- **并且** 在规范化后使用大小写敏感相等进行比较

#### 场景：处理需求重命名

- **当** 重命名需求时
- **那么** 使用特殊的 `## RENAMED Requirements` 部分
- **并且** 明确指定旧名称和新名称：
  ```markdown
  ## RENAMED Requirements
  - FROM: `### Requirement: Old Name`
  - TO: `### Requirement: New Name`
  ```
- **并且** 如果内容也发生变化，使用新标题包含在 MODIFIED 下

#### 场景：验证标题唯一性

- **当** 创建或修改需求时
- **那么** 确保规格内不存在重复标题
- **并且** 验证工具应将重复标题标记为错误

### 需求：变更存储约定

变更提案应仅存储对规格的添加、修改和删除，而不是完整的未来状态。

#### 场景：创建带添加的变更提案

- **当** 创建添加新需求的变更提案时
- **那么** 仅在 `## ADDED Requirements` 下包含新需求
- **并且** 每个需求应包含其完整内容
- **并且** 使用标准和结构化格式表示需求和场景

#### 场景：创建带修改的变更提案

- **当** 创建修改现有需求的变更提案时
- **那么** 在 `## MODIFIED Requirements` 下包含修改后的需求
- **并且** 使用与当前规格中相同的标题文本（规范化后）
- **并且** 包含完整的修改后需求（不是 diff）
- **并且** 可选择使用内联注释如 `← (was X)` 标注发生了什么变化

#### 场景：创建带删除的变更提案

- **当** 创建删除需求的变更提案时
- **那么** 在 `## REMOVED Requirements` 下列出它们
- **并且** 使用规范化标题文本进行标识
- **并且** 包含删除原因
- **并且** 如果适用，记录任何迁移路径

`changes/[name]/specs/` 目录应包含：
- 仅显示更改的增量文件
- ADDED、MODIFIED、REMOVED 和 RENAMED requirements 的部分
- 用于需求标识的规范化标题匹配
- 使用结构化格式的完整需求
- 每个需求的清晰变更类型指示

#### 场景：使用标准输出符号

- **当** 在 CLI 输出中显示增量操作时
- **那么** 使用以下标准符号：
  - `+` 用于 ADDED（绿色）
  - `~` 用于 MODIFIED（黄色）
  - `-` 用于 REMOVED（红色）
  - `→` 用于 RENAMED（青色）

### 需求：归档过程增强

归档过程应使用基于标题的匹配将增量变更程序化地应用到当前规格。

#### 场景：归档带增量的变更

- **当** 归档已完成的变更时
- **那么** archive 命令应：
  1. 首先解析 RENAMED 部分并应用重命名
  2. 解析 REMOVED 部分并通过规范化标题匹配删除
  3. 解析 MODIFIED 部分并通过规范化标题匹配替换（如果重命名则使用新名称）
  4. 解析 ADDED 部分并追加新需求
- **并且** 验证所有 MODIFIED/REMOVED 标题存在于当前规格中
- **并且** 验证 ADDED 标题尚不存在
- **并且** 在主 specs/ 目录中生成更新的规格

#### 场景：处理归档期间的冲突

- **当** 增量变更与当前规格状态冲突时
- **那么** archive 命令应报告具体冲突
- **并且** 在继续之前需要手动解决
- **并且** 提供解决冲突的清晰指导

### 需求：提案格式

提案应明确文档化所有变更，包含清晰的从/到比较。

#### 场景：记录变更

- **当** 记录发生了什么变更时
- **那么** 提案应明确描述每个变更：

```markdown
**[Section or Behavior Name]**
- From: [current state/requirement]
- To: [future state/requirement]
- Reason: [why this change is needed]
- Impact: [breaking/non-breaking, who's affected]
```

这种明确的格式补偿了没有内联 diff，并确保审阅者准确理解将要更改的内容。

### 需求：变更审查

系统应支持多种审查提议变更的方法。

#### 场景：审查变更

- **当** 审查提议的变更时
- **那么** 审阅者可以使用：
- 提交变更时的 GitHub PR diff 视图
- 命令行：`diff -u specs/[capability]/spec.md changes/[name]/specs/[capability]/spec.md`
- 任何比较当前与未来状态的视觉 diff 工具

### 需求：结构化格式 adoption

行为规格应采用 `### Requirement:` 和 `#### Scenario:` 标题作为默认值。

#### 场景：对行为使用结构化标题

- **当** 记录行为需求时
- **那么** 使用 `### Requirement:` 表示需求
- **并且** 使用 `#### Scenario:` 表示场景，带粗体 WHEN/THEN/AND 关键字

### 需求：动词-名词 CLI 命令结构
OpenSpec CLI 设计应使用动词作为顶级命令，名词作为参数或标志用于范围界定。

#### 场景：动词优先命令发现

- **当** 用户运行像 `openspec list` 这样的命令时
- **那么** 动词清楚传达动作
- **并且** 名词通过标志或参数（例如 `--changes`、`--specs`）细化范围

#### 场景：名词命令的向后兼容性

- **当** 用户运行名词前缀命令如 `openspec spec ...` 或 `openspec change ...` 时
- **那么** CLI 应继续支持至少一个版本
- **并且** 显示指向动词优先替代方案的弃用警告

#### 场景：消歧指导

- **当** 项目名称在变更和规格之间模糊时
- **那么** `openspec show` 和 `openspec validate` 应接受 `--type spec|change`
- **并且** 帮助文本应清楚记录这一点

## 核心原则

系统应遵循以下原则：
- 规格反映当前构建和部署的内容
- 变更包含应更改内容的提案
- AI 驱动文档过程
- 规格是保持与已部署代码同步的活文档

## 变更生命周期

变更过程应遵循以下状态：

1. **提案**：AI 创建带有未来状态规格和明确提案的变更
2. **审查**：人类审查提案和未来状态
3. **批准**：变更被批准用于实现
4. **实现**：按照 tasks.md 检查清单执行（可跨越多个 PR）
5. **部署**：变更部署到生产环境
6. **更新**：`specs/` 中的规格更新以匹配已部署的现实
7. **归档**：变更移动到 `archive/YYYY-MM-DD-[name]/`

## 查看变更

### 变更审查

系统应支持多种审查提议变更的方法。

#### 场景：审查变更

- **当** 审查提议的变更时
- **那么** 审阅者可以使用：
- 提交变更时的 GitHub PR diff 视图
- 命令行：`diff -u specs/[capability]/spec.md changes/[name]/specs/[capability]/spec.md`
- 任何比较当前与未来状态的视觉 diff 工具

系统依赖工具生成 diff，而不是存储它们。

## 能力命名

能力应使用：
- 动词-名词模式（例如 `user-auth`、`payment-capture`）
-  hyphenated 小写名称
- 单一关注点（每个能力一个职责）
- 不嵌套（`specs/` 下扁平结构）

## 何时变更需要提案

以下情况应创建提案：
- 新功能或能力
- 对现有行为的破坏性变更
- 架构或模式变更
- 改变行为的性能优化
- 影响访问模式的安全更新

以下情况不需要提案：
- 恢复预期行为的 bug 修复
- 拼写错误或格式修复
- 非破坏性依赖更新
- 为现有行为添加测试
- 文档澄清

## 为什么采用这种方法

干净的未来状态存储提供：
- **可读性**：没有 diff 语法污染
- **AI 兼容性**：AI 工具理解的标准 markdown
- **简单性**：不需要特殊的解析或处理
- **工具无关**：任何 diff 工具都可以显示更改
- **清晰的意图**：明确的提案记录推理

结构化格式添加：
- **视觉一致性**：需求和场景前缀使部分立即可识别
- **可解析性**：一致的结构支持工具和自动化
- **渐进 adoption**：现有规格可以增量迁移
