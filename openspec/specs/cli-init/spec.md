# CLI Init 规范

## 目的

`openspec init` 命令应在任何项目中创建完整的 OpenSpec 目录结构，实现对 OpenSpec 约定的即时采用，并支持多种 AI 编码助手。

## 需求

### 需求：进度指示器

命令应在初始化过程中显示进度指示器，以提供关于每个步骤的清晰反馈。

#### 场景：显示初始化进度

- **当** 执行初始化步骤时
- **那么** 在后台静默验证环境（无输出，除非有错误）
- **并且** 使用 ora 旋转动画显示进度：
  - 显示旋转动画："⠋ 创建 OpenSpec 结构..."
  - 然后成功："✔ OpenSpec 结构已创建"
  - 显示旋转动画："⠋ 配置 AI 工具..."
  - 然后成功："✔ AI 工具已配置"

### 需求：目录创建

命令应创建带有配置文件的 OpenSpec 目录结构。

#### 场景：创建 OpenSpec 结构

- **当** 执行 `openspec init` 时
- **那么** 创建以下目录结构：
```
openspec/
├── config.yaml
├── specs/
└── changes/
    └── archive/
```

### 需求：AI 工具配置

命令应使用可搜索的多选体验为 AI 编码助手配置技能和斜杠命令。

#### 场景：提示选择 AI 工具

- **当** 以交互方式运行时
- **那么** 显示带有 OpenSpec 徽标的动画欢迎屏幕
- **并且** 呈现显示所有可用工具的可搜索多选列表
- **并且** 用"（已配置 ✓）"标记已配置的工具有效
- **并且** 预选已配置的工具有效
- **并且** 将已配置的工具排在列表前面
- **并且** 允许通过键入进行筛选

#### 场景：选择要配置的工具

- **当** 用户选择工具并确认时
- **那么** 在 `.<tool>/skills/` 目录中为每个选定的工具生成技能
- **并且** 在 `.<tool>/commands/opsx/` 目录中为每个选定的工具生成斜杠命令
- **并且** 使用默认 schema 设置创建 `openspec/config.yaml`

### 需求：交互模式
命令应为 AI 工具选择提供带有清晰导航说明的交互式菜单。

#### 场景：显示交互式菜单

- **当** 以全新或扩展模式运行时
- **那么** 呈现一个循环选择菜单，允许用户使用空格键切换工具，并用回车键查看选择
- **并且** 当在未选中的高亮可选项上按回车时，自动将其添加到选择中，然后进入查看阶段，以便高亮的工具被配置
- **并且** 用"（已配置）"标记已配置的工具，同时保持禁用选项标记为"即将推出"
- **并且** 在扩展模式下将提示文字改为"您想添加或刷新哪些 AI 工具？"
- **并且** 显示内联说明，澄清空格键切换工具，回车键在高亮工具进入查看阶段之前选择它

### 需求：安全检查

命令应执行安全检查以防止覆盖现有结构并确保适当的权限。

#### 场景：检测现有初始化

- **当** `openspec/` 目录已存在时
- **那么** 通知用户 OpenSpec 已初始化，跳过重新创建基础结构，并进入扩展模式
- **并且** 继续进行 AI 工具选择步骤，以便可以配置其他工具
- **并且** 仅当用户拒绝添加任何 AI 工具时才显示现有初始化错误消息

### 需求：成功输出

命令应在成功初始化后提供清晰的、可操作的next steps。

#### 场景：显示成功消息

- **当** 初始化成功完成时
- **那么** 显示分类摘要：
  - "已创建：<工具>" 用于新配置的工具
  - "已刷新：<工具>" 用于已更新且已配置的工具有效
  - 生成的技能和命令计数
- **并且** 显示入门部分，其中包含：
  - `/opsx:new` - 开始新的变更
  - `/opsx:continue` - 创建下一个产物
  - `/opsx:apply` - 实施任务
- **并且** 显示文档和反馈链接

#### 场景：显示重启说明

- **当** 初始化成功完成且工具已被创建或刷新时
- **那么** 显示重启 IDE 的说明，以使斜杠命令生效

### 需求：退出代码

命令应使用一致的退出代码来表示不同的失败模式。

#### 场景：返回退出代码

- **当** 命令完成时
- **那么** 返回适当的退出代码：
  - 0：成功
  - 1：常规错误（包括 OpenSpec 目录已存在时）
  - 2：权限不足（保留供将来使用）
  - 3：用户取消操作（保留供将来使用）

### 需求：额外的 AI 工具初始化
`openspec init` 应允许用户在初始设置后为新的 AI 编码助手添加配置文件。

#### 场景：在初始设置后配置额外的工具

- **假设** `openspec/` 目录已存在且至少存在一个 AI 工具文件
- **当** 用户运行 `openspec init` 并选择其他受支持的 AI 工具时
- **那么** 以与首次初始化相同的方式使用 OpenSpec 标记生成该工具的配置文件
- **并且** 除需要刷新的管理部分外，保持现有工具配置文件不变
- **并且** 以代码 0 退出，并显示突出显示新添加的工具文件的成功摘要

### 需求：成功输出增强
`openspec init` 应在初始化或扩展模式完成时总结工具操作。

#### 场景：显示工具摘要

- **当** 命令成功完成时
- **那么** 显示按创建、刷新或跳过（包括已配置跳过）分类的工具摘要
- **并且** 使用所选工具的名称个性化"next steps"标题，默认使用通用标签

### 需求：退出代码调整
`openspec init` 应将没有新原生工具选择的扩展模式视为成功的刷新。

#### 场景：允许空扩展运行

- **当** OpenSpec 已初始化且用户未选择其他本地支持的工具时
- **那么** 成功完成，无需额外的工具设置
- **并且** 保留现有的 OpenSpec 结构和配置文件
- **并且** 以代码 0 退出

### 需求：非交互式模式

命令应通过命令行选项支持非交互式操作。

#### 场景：非交互式选择所有工具

- **当** 使用 `--tools all` 运行
- **那么** 自动选择每个可用的 AI 工具，不提示
- **并且** 继续进行技能和命令生成

#### 场景：非交互式选择特定工具

- **当** 使用 `--tools claude,cursor` 运行
- **那么** 解析逗号分隔的工具 ID
- **并且** 仅针对指定的工具生成技能和命令

#### 场景：非交互式跳过工具配置

- **当** 使用 `--tools none` 运行
- **那么** 仅创建 openspec 目录结构
- **并且** 跳过技能和命令生成
- **并且** 仅在满足配置创建条件时创建配置

#### 场景：无效的工具规范

- **当** 使用 `--tools invalid-tool` 运行
- **那么** 以退出代码 1 失败
- **并且** 显示列出可用值的错误（`all`、`none` 和支持的工具 ID）

#### 场景：将保留值与工具 ID 组合

- **当** 使用 `--tools all,claude` 或 `--tools none,cursor` 运行
- **那么** 以退出代码 1 失败
- **并且** 显示解释保留值不能与特定工具 ID 组合的错误

#### 场景：在非交互式模式下缺少 --tools

- **假设** 在非交互式执行中提示不可用
- **当** 用户在没有 `--tools` 的情况下运行 `openspec init`
- **那么** 以退出代码 1 失败
- **并且** 指示使用 `--tools all`、`--tools none` 或显式工具 ID

### 需求：技能生成

命令应为选定的 AI 工具生成 Agent Skills。

#### 场景：为工具生成技能

- **当** 在初始化期间选择了一个工具时
- **那么** 在 `.<tool>/skills/` 下创建 9 个技能目录：
  - `openspec-explore/SKILL.md`
  - `openspec-new-change/SKILL.md`
  - `openspec-continue-change/SKILL.md`
  - `openspec-apply-change/SKILL.md`
  - `openspec-ff-change/SKILL.md`
  - `openspec-verify-change/SKILL.md`
  - `openspec-sync-specs/SKILL.md`
  - `openspec-archive-change/SKILL.md`
  - `openspec-bulk-archive-change/SKILL.md`
- **并且** 每个 SKILL.md 应包含带有 name 和 description 的 YAML frontmatter
- **并且** 每个 SKILL.md 应包含技能指令

### 需求：斜杠命令生成

命令应仅为具有注册命令适配器的选定工具生成 opsx 斜杠命令，同时保持适配器less工具的技能生成有效。

#### 场景：为具有注册适配器的工具生成斜杠命令

- **当** 在初始化期间选择了具有注册命令适配器的工具时
- **那么** 使用该工具的命令适配器创建 9 个斜杠命令文件：
  - `/opsx:explore`
  - `/opsx:new`
  - `/opsx:continue`
  - `/opsx:apply`
  - `/opsx:ff`
  - `/opsx:verify`
  - `/opsx:sync`
  - `/opsx:archive`
  - `/opsx:bulk-archive`
- **并且** 使用工具特定的路径约定（例如，`.claude/commands/opsx/` 用于 Claude）
- **并且** 包含工具特定的 frontmatter 格式

#### 场景：所选工具没有命令适配器

- **假设** 所选工具配置了 `skillsDir` 但没有注册的命令适配器
- **当** 初始化包括命令生成时
- **那么** 该工具的技能生成仍应保持有效
- **并且** 该工具的命令文件生成应被跳过
- **并且** 命令输出应包括 `Commands skipped for: <tool-id> (no adapter)`

#### 场景：Kimi CLI 跳过命令文件生成

- **当** 用户在初始化期间选择 Kimi CLI 时
- **那么** OpenSpec 应将其视为支持的工具，配置为 `skillsDir: '.kimi'`
- **并且** 命令文件生成应被跳过，因为没有注册 Kimi 适配器

### 需求：配置文件生成

命令应创建带有 schema 设置的 OpenSpec 配置文件。

#### 场景：创建 config.yaml

- **当** 初始化完成时
- **并且** config.yaml 不存在
- **那么** 使用默认 schema 设置创建 `openspec/config.yaml`
- **并且** 在输出中显示配置位置

#### 场景：保留现有的 config.yaml

- **当** 初始化以扩展模式运行时
- **并且** `openspec/config.yaml` 已存在
- **那么** 保留现有配置文件
- **并且** 在输出中显示"（存在）"指示器

### 需求：实验性命令别名

命令应维护与实验性命令的向后兼容性。

#### 场景：运行 openspec experimental

- **当** 用户运行 `openspec experimental` 时
- **那么** 委托给 `openspec init`
- **并且** 该命令应从帮助输出中隐藏

## 为什么

手动创建 OpenSpec 结构容易出错且会增加采用摩擦。标准化的 init 命令确保：
- 所有项目之间的一致结构
- 始终包含正确的 AI 说明文件
- 新项目的快速入门
- 从一开始就建立清晰的约定
