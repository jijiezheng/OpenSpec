# opsx-verify-skill 规范

## 目的
定义 `/opsx:verify` 行为，用于根据变更产物评估实现的完整性、正确性和一致性。

## 需求

### 需求：Verify 技能调用
系统应提供 `/opsx:verify` 技能，用于验证实现是否符合变更产物。

#### 场景：提供变更名称的验证
- **当** 代理执行 `/opsx:verify <change-name>` 时
- **那么** 代理验证该特定变更的实现
- **并且** 生成验证报告

#### 场景：没有变更名称的验证
- **当** 代理执行 `/opsx:verify` 而不提供变更名称时
- **那么** 代理提示用户从可用变更中选择
- **并且** 仅显示有实现任务的变更

#### 场景：变更没有任务
- **当** 选中的变更没有 tasks.md 或任务为空时
- **那么** 代理报告"No tasks to verify"
- **并且** 建议运行 `/opsx:continue` 创建任务

### 需求：完整性验证
代理应验证所有必需的工作已完成。

#### 场景：任务完成检查
- **当** 验证完整性时
- **那么** 代理读取 tasks.md
- **并且** 计算标记为 `- [x]`（完成）与 `- [ ]`（未完成）的任务
- **并且** 报告完成状态，列出具体的未完成任务

#### 场景：规格覆盖检查
- **当** 验证完整性时
- **并且** 增量规格存在于 `openspec/changes/<name>/specs/` 中
- **那么** 代理从增量规格中提取所有需求
- **并且** 在代码库中搜索每个需求的实现
- **并且** 报告哪些需求似乎有实现，哪些缺失

#### 场景：所有任务完成
- **当** 所有任务都标记为完成时
- **那么** 报告"Tasks: N/N complete"
- **并且** 将完整性维度标记为通过

#### 场景：发现未完成的任务
- **当** 一些任务未完成时
- **那么** 报告"Tasks: X/N complete"
- **并且** 列出每个未完成的任务
- **并且** 标记为 CRITICAL 问题
- **并且** 建议："Complete remaining tasks or mark as done if already implemented"

### 需求：正确性验证
代理应验证实现是否符合规格。

#### 场景：需求实现映射
- **当** 验证正确性时
- **那么** 对于增量规格中的每个需求：
  - 在代码库中搜索实现
  - 识别相关文件和行号
  - 评估实现是否满足需求

#### 场景：场景覆盖检查
- **当** 验证正确性时
- **那么** 对于增量规格中的每个场景：
  - 检查代码中是否处理了场景的条件
  - 检查是否有覆盖场景的测试
  - 报告覆盖状态

#### 场景：实现符合规格
- **当** 实现似乎满足需求时
- **那么** 报告哪些文件/行实现了它
- **并且** 将需求标记为已覆盖

#### 场景：实现与规格偏离
- **当** 实现存在但不符合规格意图时
- **那么** 报告偏离作为 WARNING
- **并且** 解释有什么不同
- **并且** 建议：要么更新实现，要么更新规格以匹配现实

#### 场景：缺失实现
- **当** 找不到需求实现时
- **那么** 报告为 CRITICAL 问题
- **并且** 建议："Implement requirement X" 并提供需要的指导

### 需求：一致性验证
代理应验证实现是否合理且遵循设计决策。

#### 场景：Design.md 遵守检查
- **当** 验证一致性时
- **并且** 变更存在 design.md 时
- **那么** 从 design.md 提取关键决策
- **并且** 验证实现是否遵循这些决策
- **并且** 报告任何偏差

#### 场景：没有 design.md
- **当** 验证一致性时
- **并且** 不存在 design.md 时
- **那么** 跳过设计遵守检查
- **并且** 注明"No design.md to verify against"

#### 场景：遵循设计决策
- **当** 实现遵循设计决策时
- **那么** 报告已确认
- **并且** 引用代码证据

#### 场景：违反设计决策
- **当** 实现与设计决策矛盾时
- **那么** 报告为 WARNING
- **并且** 解释矛盾
- **并且** 建议：要么更新实现，要么更新 design.md

#### 场景：代码模式一致性
- **当** 验证一致性时
- **那么** 检查新代码是否遵循现有项目模式
- **并且** 将任何显著偏差标记为建议

### 需求：验证报告格式
代理应生成结构化的、优先级的报告。

#### 场景：报告摘要
- **当** 验证完成时
- **那么** 显示摘要记分卡：
  ```text
  ## Verification Report: <change-name>

  ### Summary
  | Dimension    | Status   |
  |--------------|----------|
  | Completeness | X/Y      |
  | Correctness  | X/Y      |
  | Coherence    | Followed |
  ```

#### 场景：问题优先级
- **当** 发现问题时
- **那么** 按优先级顺序分组和显示：
  1. CRITICAL - 归档前必须修复（缺失实现、任务未完成）
  2. WARNING - 应该修复（与规格/设计偏离、缺失测试）
  3. SUGGESTION - 最好修复（模式不一致、次要改进）

#### 场景：可操作的建议
- **当** 报告问题时
- **那么** 包含具体的、可操作的修复建议
- **并且** 在适用时引用相关文件和行号
- **并且** 避免模糊建议如"consider reviewing"

#### 场景：所有检查通过
- **当** 所有维度都没有发现问题时
- **那么** 显示：
  ```text
  All checks passed. Ready for archive.
  ```

#### 场景：发现关键问题
- **当** 存在 CRITICAL 问题时
- **那么** 显示：
  ```text
  X critical issue(s) found. Fix before archiving.
  ```
- **并且** 不建议运行归档

#### 场景：仅有警告/建议
- **当** 没有 CRITICAL 问题但存在警告时
- **那么** 显示：
  ```text
  No critical issues. Y warning(s) to consider.
  Ready for archive (with noted improvements).
  ```

### 需求：灵活的产物处理
代理应优雅地处理产物完成度不同的变更。

#### 场景：最小变更（仅有任务）
- **当** 变更仅有 tasks.md 时
- **那么** 仅验证任务完成
- **并且** 跳过规格和设计检查
- **并且** 注明跳过了哪些检查

#### 场景：有规格但没有设计的变更
- **当** 变更有 tasks.md 和增量规格但没有 design.md 时
- **那么** 验证完整性和正确性
- **并且** 跳过设计遵守
- **并且** 仍然根据项目模式检查代码一致性

#### 场景：完整变更（所有产物）
- **当** 变更有 proposal、design、specs 和 tasks 时
- **那么** 执行所有验证检查
- **并且** 交叉引用产物以确保一致性
