# OPSX 工作流

> 欢迎在 [Discord](https://discord.gg/YctCnvvshC) 上反馈意见。

## 这是什么？

OPSX 现在是 OpenSpec 的标准工作流。

这是一种**流动的、迭代的工作流**，用于 OpenSpec 变更。不再有僵硬的阶段 — 只有你可以随时执行的动作。

## 为什么存在

旧版 OpenSpec 工作流可用，但它是**锁定的**：

- **指令硬编码** — 埋在 TypeScript 中，无法更改
- **全有或全无** — 一个大命令创建所有内容，无法测试单个部分
- **固定结构** — 对每个人都一样的工作流，无法自定义
- **黑箱** — 当 AI 输出不好时，你无法调整提示

**OPSX 打开了它。** 现在任何人都可以：

1. **实验指令** — 编辑模板，看看 AI 是否做得更好
2. **细粒度测试** — 独立验证每个产物的指令
3. **自定义工作流** — 定义你自己的产物和依赖
4. **快速迭代** — 更改模板、立即测试、无需重建

```
旧版工作流：                      OPSX：
┌────────────────────────┐           ┌────────────────────────┐
│  包中硬编码            │           │  schema.yaml           │◄── 你可以编辑这个
│  （无法更改）          │           │  templates/*.md        │◄── 或这个
│        ↓               │           │        ↓               │
│  等待新版本            │           │  即时生效               │
│        ↓               │           │        ↓               │
│  希望它更好            │           │  自己测试               │
└────────────────────────┘           └────────────────────────┘
```

**这适合所有人：**
- **团队** — 创建匹配你实际工作方式的工作流
- **高级用户** — 调整提示以从 AI 获得更好的输出
- **OpenSpec 贡献者** — 无需发布即可尝试新方法

我们都在学习什么最有效。OPSX 让我们一起学习。

## 用户体验

**线性工作流的问题：**
你"处于规划阶段"，然后"处于实施阶段"，然后"完成"。但真实工作不是这样的。你实施了一些东西，发现设计错了，需要更新规格，继续实施。线性阶段违背了工作实际发生的方式。

**OPSX 方法：**
- **动作而非阶段** — 创建、实施、更新、归档 — 随时可以做任何一项
- **依赖是赋能者** — 它们展示什么是可能的，而非下一步必须做什么

```
  提案 ──→ 规格 ──→ 设计 ──→ 任务 ──→ 实施
```

## 设置

```bash
# 确保已安装 openspec — 技能会自动生成
openspec init
```

这会在 `.claude/skills/`（或等效目录）创建技能，AI 编码助手会自动检测。

默认情况下，OpenSpec 使用 `core` 工作流配置文件（`propose`、`explore`、`apply`、`sync`、`archive`）。如果你想要扩展的工作流命令（`new`、`continue`、`ff`、`verify`、`bulk-archive`、`onboard`），用 `openspec config profile` 配置，然后用 `openspec update` 应用。

在设置过程中，系统会提示你创建**项目配置**（`openspec/config.yaml`）。这是可选的，但建议创建。

## 项目配置

项目配置让你设置默认值，并将项目特定的上下文注入到所有产物中。

### 创建配置

配置在 `openspec init` 期间创建，或手动创建：

```yaml
# openspec/config.yaml
schema: spec-driven

context: |
  技术栈：TypeScript、React、Node.js
  API 规范：RESTful、JSON 响应
  测试：Vitest 单元测试、Playwright 端到端测试
  代码风格：ESLint + Prettier、严格 TypeScript

rules:
  proposal:
    - 包含回滚计划
    - 识别受影响的团队
  specs:
    - 使用 Given/When/Then 格式编写场景
  design:
    - 复杂流程包含序列图
```

### 配置字段

| 字段 | 类型 | 描述 |
|-------|------|-------------|
| `schema` | 字符串 | 新变更的默认 schema（例如 `spec-driven`） |
| `context` | 字符串 | 注入到所有产物指令中的项目上下文 |
| `rules` | 对象 | 按产物 ID 键入的每产物规则 |

### 工作原理

**Schema 优先级**（从高到低）：
1. CLI 标志（`--schema <name>`）
2. 变更元数据（变更目录中的 `.openspec.yaml`）
3. 项目配置（`openspec/config.yaml`）
4. 默认值（`spec-driven`）

**上下文注入：**
- 上下文被前置到每个产物的指令
- 用 `<context>...</context>` 标签包裹
- 帮助 AI 理解你项目的约定

**规则注入：**
- 规则仅注入到匹配的产物
- 用 `<rules>...</rules>` 标签包裹
- 出现在上下文之后、模板之前

### 按 Schema 的产物 ID

**spec-driven**（默认）：
- `proposal` — 变更提案
- `specs` — 规格说明
- `design` — 技术设计
- `tasks` — 实施任务

### 配置验证

- 规则中未知的产物 ID 会产生警告
- Schema 名称根据可用 schema 验证
- 上下文有 50KB 大小限制
- 报告无效 YAML 及行号

### 故障排除

**"规则中未知的产物 ID：X"**
- 检查产物 ID 是否与你的 schema 匹配（见上文列表）
- 运行 `openspec schemas --json` 查看每个 schema 的产物 ID

**配置未应用：**
- 确保文件在 `openspec/config.yaml`（不是 `.yml`）
- 用验证器检查 YAML 语法
- 配置更改立即生效（无需重启）

**上下文太大：**
- 上下文限制为 50KB
- 总结或链接到外部文档

## 命令

| 命令 | 功能 |
|---------|--------------|
| `/opsx:propose` | 一步创建变更并生成规划产物（默认快速路径） |
| `/opsx:explore` | 思考想法、调查问题、澄清需求 |
| `/opsx:new` | 启动新的变更脚手架（扩展工作流） |
| `/opsx:continue` | 创建下一个产物（扩展工作流） |
| `/opsx:ff` | 快进规划产物（扩展工作流） |
| `/opsx:apply` | 实施任务，按需更新产物 |
| `/opsx:verify` | 根据产物验证实施（扩展工作流） |
| `/opsx:sync` | 将增量规格同步到主分支（默认工作流，可选） |
| `/opsx:archive` | 归档完成时 |
| `/opsx:bulk-archive` | 归档多个已完成的变更（扩展工作流） |
| `/opsx:onboard` | 端到端变更的引导式演练（扩展工作流） |

## 使用

### 探索一个想法
```
/opsx:explore
```
思考想法、调查问题、比较选项。不需要结构 — 只是一个思考伙伴。当洞察清晰时，过渡到 `/opsx:propose`（默认）或 `/opsx:new`/`/opsx:ff`（扩展）。

### 开始一个新的变更
```
/opsx:propose
```
创建变更并生成实施前所需的规划产物。

如果你已启用扩展工作流，可以改用：

```text
/opsx:new        # 仅脚手架
/opsx:continue   # 一次创建一个产物
/opsx:ff         # 一次创建所有规划产物
```

### 创建产物
```
/opsx:continue
```
显示基于依赖关系准备创建的内容，然后创建一个产物。重复使用以增量构建你的变更。

```
/opsx:ff add-dark-mode
```
一次创建所有规划产物。当你清楚自己要构建的内容时使用。

### 实施（流动的部分）
```
/opsx:apply
```
处理任务，逐个检查。如果你同时处理多个变更，可以运行 `/opsx:apply <name>`；否则它应该从对话中推断，如果无法判断会提示你选择。

### 完成
```
/opsx:archive   # 完成后移动到归档（如果需要会提示同步规格）
```

## 何时更新 vs 重新开始

你随时可以在实施前编辑提案或规格。但什么时候精炼变成了"这是不同的工作"？

### 提案捕获的内容

提案定义三件事：
1. **意图** — 你要解决什么问题？
2. **范围** — 什么在界限内/外？
3. **方法** — 你将如何解决它？

问题是：哪个变了，变了多少？

### 在以下情况下更新现有变更：

**意图相同、执行优化**
- 你发现了你没有考虑到的边缘情况
- 方法需要调整但目标不变
- 实施揭示设计稍微不对

**范围缩小**
- 你意识到完整范围太大，想先发布 MVP
- "添加暗色模式" → "添加暗色模式切换（v2 中的系统偏好）"

**学习驱动的修正**
- 代码库结构与你想象的不同
- 依赖项不如预期那样工作
- "使用 CSS 变量" → "改用 Tailwind 的 dark: 前缀"

### 在以下情况下开始新变更：

**意图根本改变**
- 问题本身现在不同了
- "添加暗色模式" → "添加包含自定义颜色、字体、间距的综合主题系统"

**范围爆炸**
- 变更增长到本质上不同的工作
- 原始提案在更新后会无法辨认
- "修复登录 bug" → "重写 auth 系统"

**原始可完成**
- 原始变更可以标记为"完成"
- 新工作独立存在，不是精炼
- 完成"添加暗色模式 MVP" → 归档 → 新变更"增强暗色模式"

### 启发式方法

```
                         ┌─────────────────────────────────────┐
                         │     这是相同的工作吗？             │
                         └──────────────┬──────────────────────┘
                                        │
                     ┌──────────────────┼──────────────────┐
                     │                  │                  │
                     ▼                  ▼                  ▼
              意图相同？           超过 50% 重叠？      原始变更
              问题相同？            范围相同？           能否独立"完成"？
                     │                  │                 │
           ┌────────┴────────┐  ┌──────┴──────┐   ┌───────┴───────┐
           │                 │  │             │   │               │
          是               否 是            否  否              是
           │                 │  │             │   │               │
           ▼                 ▼  ▼             ▼   ▼               ▼
        更新              新建 更新          新建 更新           新建
```

| 测试 | 更新 | 新变更 |
|------|--------|------------|
| **身份** | "同样的事，精炼后" | "不同的工作" |
| **范围重叠** | >50% 重叠 | <50% 重叠 |
| **完成** | 没有变更无法"完成" | 可以完成原始的，新工作独立存在 |
| **故事** | 更新链讲述一致的故事 | 补丁会造成更多混淆而非澄清 |

### 原则

> **更新保留上下文。新变更提供清晰度。**
>
> 当你的思考历史有价值时选择更新。
> 当重新开始比补丁更清晰时选择新变更。

把它想象成 git 分支：
- 在同一个功能上继续提交
- 当真正是新工作时开始新分支
- 有时合并部分功能并为第二阶段重新开始

## 有什么不同？

| | 旧版（`/openspec:proposal`） | OPSX（`/opsx:*`） |
|---|---|---|
| **结构** | 一个大的提案文档 | 具有依赖关系的离散产物 |
| **工作流** | 线性阶段：计划 → 实施 → 归档 | 流动动作 — 随时做任何事 |
| **迭代** | 回头很尴尬 | 在学习时更新产物 |
| **自定义** | 固定结构 | Schema 驱动（定义你自己的产物） |

**关键洞察：** 工作不是线性的。OPSX 停止假装它是。

## 架构深入

本节解释 OPSX 在幕后如何工作，以及它与旧版工作流的比较。本节中的示例使用扩展命令集（`new`、`continue` 等）；默认的 `core` 用户可以将相同流程映射到 `propose → apply → sync → archive`。

### 理念：阶段 vs 动作

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         旧版工作流                                          │
│                    （阶段锁定、全有或全无）                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐             │
│   │   规划       │ ───► │    实施      │ ───► │    归档      │             │
│   │    阶段      │      │    阶段      │      │    阶段      │             │
│   └──────────────┘      └──────────────┘      └──────────────┘             │
│         │                     │                     │                       │
│         ▼                     ▼                     ▼                       │
│   /openspec:proposal   /openspec:apply      /openspec:archive              │
│                                                                             │
│   • 一次创建所有产物                                          │
│   • 实施期间无法回头更新规格                                   │
│   • 阶段门强制线性进度                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘


┌─────────────────────────────────────────────────────────────────────────────┐
│                            OPSX 工作流                                       │
│                      （流动动作、迭代）                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│              ┌────────────────────────────────────────────┐                 │
│              │           动作（不是阶段）                │                 │
│              │                                            │                 │
│              │   new ◄──► continue ◄──► apply ◄──► archive │                 │
│              │    │          │           │           │    │                 │
│              │    └──────────┴───────────┴───────────┘    │                 │
│              │              任意顺序                      │                 │
│              └────────────────────────────────────────────┘                 │
│                                                                             │
│   • 一次创建一个产物或快进                                        │
│   • 实施期间更新规格/设计/任务                                    │
│   • 依赖启用进度，阶段不存在                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 组件架构

**旧版工作流**使用 TypeScript 中的硬编码模板：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      旧版工作流组件                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   硬编码模板（TypeScript 字符串）                                           │
│                    │                                                        │
│                    ▼                                                        │
│   工具特定配置器/适配器                                                      │
│                    │                                                        │
│                    ▼                                                        │
│   生成的命令文件（.claude/commands/openspec/*.md）                          │
│                                                                             │
│   • 固定结构，无产物感知                                                    │
│   • 变更需要代码修改 + 重建                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**OPSX** 使用外部 schema 和依赖图引擎：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         OPSX 组件                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   Schema 定义（YAML）                                                        │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  name: spec-driven                                                  │   │
│   │  artifacts:                                                         │   │
│   │    - id: proposal                                                   │   │
│   │      generates: proposal.md                                         │   │
│   │      requires: []              ◄── 依赖                             │   │
│   │    - id: specs                                                      │   │
│   │      generates: specs/**/*.md  ◄── Glob 模式                         │   │
│   │      requires: [proposal]      ◄── 在提案后启用                       │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                    │                                                        │
│                    ▼                                                        │
│   产物图引擎                                                                │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  • 拓扑排序（依赖排序）                                               │   │
│   │  • 状态检测（文件系统存在性）                                         │   │
│   │  • 丰富指令生成（模板 + 上下文）                                      │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                    │                                                        │
│                    ▼                                                        │
│   技能文件（.claude/skills/openspec-*/SKILL.md）                            │
│                                                                             │
│   • 跨编辑器兼容（Claude Code、Cursor、Windsurf）                          │
│   • 技能查询 CLI 获取结构化数据                                            │
│   • 完全可通过 schema 文件自定义                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 依赖图模型

产物形成有向无环图（DAG）。依赖是**赋能者**，不是门：

```
                              proposal
                             （根节点）
                                   │
                     ┌─────────────┴─────────────┐
                     │                           │
                     ▼                           ▼
                  specs                       design
               (requires:                  (requires:
                proposal)                   proposal)
                     │                           │
                     └─────────────┬─────────────┘
                                   │
                                   ▼
                                tasks
                            (requires:
                            specs, design)
                                   │
                                   ▼
                           ┌──────────────┐
                           │   实施阶段   │
                           │ (requires:   │
                           │  tasks)      │
                           └──────────────┘
```

**状态转换：**

```
   BLOCKED ────────────────► READY ────────────────► DONE
      │                        │                       │
   缺少                      所有依赖                 文件存在于
   依赖                      都是 DONE               文件系统
```

### 信息流

**旧版工作流** — 代理接收静态指令：

```
  用户: "/openspec:proposal"
            │
            ▼
  ┌─────────────────────────────────────────┐
  │  静态指令：                              │
  │  • 创建 proposal.md                     │
  │  • 创建 tasks.md                        │
  │  • 创建 design.md                       │
  │  • 创建 specs/<capability>/spec.md      │
  │                                         │
  │  不感知存在什么或                        │
  │  产物之间的依赖关系                      │
  └─────────────────────────────────────────┘
            │
            ▼
  代理一次创建所有产物
```

**OPSX** — 代理查询丰富上下文：

```
  用户: "/opsx:continue"
            │
            ▼
  ┌──────────────────────────────────────────────────────────────────────────┐
  │  步骤 1：查询当前状态                                                  │
  │  ┌────────────────────────────────────────────────────────────────────┐  │
  │  │  $ openspec status --change "add-auth" --json                      │  │
  │  │                                                                    │  │
  │  │  {                                                                 │  │
  │  │    "artifacts": [                                                  │  │
  │  │      {"id": "proposal", "status": "done"},                         │  │
  │  │      {"id": "specs", "status": "ready"},      ◄── 第一个就绪        │  │
  │  │      {"id": "design", "status": "ready"},                          │  │
  │  │      {"id": "tasks", "status": "blocked", "missingDeps": ["specs"]}│  │
  │  │    ]                                                               │  │
  │  │  }                                                                 │  │
  │  └────────────────────────────────────────────────────────────────────┘  │
  │                                                                          │
  │  步骤 2：获取就绪产物的丰富指令                                         │
  │  ┌────────────────────────────────────────────────────────────────────┐  │
  │  │  $ openspec instructions specs --change "add-auth" --json          │  │
  │  │                                                                    │  │
  │  │  {                                                                 │  │
  │  │    "template": "# 规格\n\n## 新增需求...",                        │  │
  │  │    "dependencies": [{"id": "proposal", "path": "...", "done": true}│  │
  │  │    "unlocks": ["tasks"]                                            │  │
  │  │  }                                                                 │  │
  │  └────────────────────────────────────────────────────────────────────┘  │
  │                                                                          │
  │  步骤 3：读取依赖 → 创建单个产物 → 显示解锁了什么                       │
  └──────────────────────────────────────────────────────────────────────────┘
```

### 迭代模型

**旧版工作流** — 迭代尴尬：

```
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │/proposal│ ──► │ /apply  │ ──► │/archive │
  └─────────┘     └─────────┘     └─────────┘
       │               │
       │               ├── "等等，设计错了"
       │               │
       │               ├── 选项：
       │               │   • 手动编辑文件（破坏上下文）
       │               │   • 放弃并重新开始
       │               │   • 推进并稍后修复
       │               │
       │               └── 没有正式的"返回"机制
       │
       └── 一次创建所有产物
```

**OPSX** — 自然迭代：

```
  /opsx:new ───► /opsx:continue ───► /opsx:apply ───► /opsx:archive
      │                │                  │
      │                │                  ├── "设计错了"
      │                │                  │
      │                │                  ▼
      │                │            只需编辑 design.md
      │                │            然后继续！
      │                │                  │
      │                │                  ▼
      │                │         /opsx:apply 从
      │                │         你停止的地方继续
      │                │
      │                └── 创建单个产物，显示解锁了什么
      │
      └── 脚手架变更，等待指示
```

### 自定义 Schema

使用 schema 管理命令创建自定义工作流：

```bash
# 从头开始创建新 schema（交互式）
openspec schema init my-workflow

# 或 fork 现有 schema 作为起点
openspec schema fork spec-driven my-workflow

# 验证你的 schema 结构
openspec schema validate my-workflow

# 查看 schema 从哪里解析（用于调试）
openspec schema which my-workflow
```

Schema 存储在 `openspec/schemas/`（项目本地，版本控制）或 `~/.local/share/openspec/schemas/`（用户全局）。

**Schema 结构：**
```
openspec/schemas/research-first/
├── schema.yaml
└── templates/
    ├── research.md
    ├── proposal.md
    └── tasks.md
```

**示例 schema.yaml：**
```yaml
name: research-first
artifacts:
  - id: research        # 在提案之前添加
    generates: research.md
    requires: []

  - id: proposal
    generates: proposal.md
    requires: [research]  # 现在依赖 research

  - id: tasks
    generates: tasks.md
    requires: [proposal]
```

**依赖图：**
```
   research ──► proposal ──► tasks
```

### 摘要

| 方面 | 旧版 | OPSX |
|--------|----------|------|
| **模板** | 硬编码 TypeScript | 外部 YAML + Markdown |
| **依赖** | 无（一次全部） | 带拓扑排序的 DAG |
| **状态** | 基于阶段的心理模型 | 文件系统存在性 |
| **自定义** | 编辑源码，重建 | 创建 schema.yaml |
| **迭代** | 阶段锁定 | 流动，编辑任何内容 |
| **编辑器支持** | 工具特定的配置器/适配器 | 单一技能目录 |

## Schema

Schema 定义存在哪些产物及其依赖。当前可用：

- **spec-driven**（默认）：proposal → specs → design → tasks

```bash
# 列出可用 schema
openspec schemas

# 查看所有 schema 及其解析来源
openspec schema which --all

# 交互式创建新 schema
openspec schema init my-workflow

# fork 现有 schema 进行定制
openspec schema fork spec-driven my-workflow

# 使用前验证 schema 结构
openspec schema validate my-workflow
```

## 提示

- 在提交变更之前使用 `/opsx:explore` 思考一个想法
- 当你知道想要什么时用 `/opsx:ff`，探索时用 `/opsx:continue`
- 在 `/opsx:apply` 期间，如果有问题 — 修复产物，然后继续
- 任务通过 `tasks.md` 中的复选框跟踪进度
- 随时检查状态：`openspec status --change "name"`
